- Regional APIs

рҹ§  System Layers
1. User Layer
- CDN вҶ’ fast static delivery
- Frontend вҶ’ user interaction
2. API Layer
- API Gateway:
- Authentication
- Routing
Backend:
- Business logic
- Aggregation
3. Performance Layer
- Cache (Redis)
- Reduce ERP calls
- Improve latency
- Search (Elastic)
- Fuzzy search
- Fast querying
4. Reliability Layer
- Idempotency
- Prevent duplicate operations
Key structure:
- idempotency_key
- status
- response
5. Integration Layer
- Kafka / Event Bus
- Async communication
- Decoupling
- ERP Systems
- SAP S/4HANA
Oracle / MSFT

рҹҢҚ Multi-Region Strategy
- Deploy APIs regionally
- Replicate events globally
Use:
- event_id
- version

вҡ–пёҸ Key Design Decisions
Why Not Direct ERP Calls?
- ERP is slow
- Not scalable
- Tight coupling
Why Event-Driven?
- Loose coupling
- Scalable
- Retry support
Why Idempotency?
- Prevent duplicate orders
- Critical in distributed systems

рҹ”Ҙ Golden Rules
- Never expose ERP directly to frontend
Always introduce:
- API layer
- Cache
- Event bus
Separate:
- Read vs Write paths

рҹҡ« When NOT to Use This Design
- Small applications
- Low traffic systems
- Single-region systems

рҹ“Ң Real-World Example

User creates order:

1. Frontend вҶ’ API
2. Backend validates + idempotency check
3. SAP creates order
4. Event emitted
5. Other systems consume
