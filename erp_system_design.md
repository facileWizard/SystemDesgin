- Users interact via a frontend
- ERP (SAP) acts as the backend

System should support:
- High traffic
- Multi-region deployment
- Reliable integrations

## рҹҢҚ Multi-Region Strategy

- Deploy APIs regionally  
- Replicate events globally  

Use:
- `event_id`
- `version`

---

## рҹ§  System Layers

### 1. User Layer
- CDN вҶ’ Fast static delivery  
- Frontend вҶ’ User interaction  

### 2. API Layer

**API Gateway:**
- Authentication  
- Routing  

**Backend:**
- Business logic  
- Aggregation  

---

### 3. Performance Layer

**Cache (Redis):**
- Reduce ERP calls  
- Improve latency  

**Search (Elastic):**
- Fuzzy search  
- Fast querying  

---

### 4. Reliability Layer

**Idempotency:**
- Prevent duplicate operations  

**Key Structure:**
- `idempotency_key`  
- `status`  
- `response`  

---

### 5. Integration Layer

**Kafka / Event Bus:**
- Async communication  
- Decoupling  

**ERP Systems:**
- SAP S/4HANA  
- Oracle / Microsoft  

---

## вҡ–пёҸ Key Design Decisions

### Why NOT Direct ERP Calls?
- ERP is slow  
- Not scalable  
- Tight coupling  

### Why Event-Driven?
- Loose coupling  
- Scalable  
- Retry support  

### Why Idempotency?
- Prevent duplicate orders  
- Critical in distributed systems  

---

## рҹ”Ҙ Golden Rules

- Never expose ERP directly to frontend  

Always introduce:
- API layer  
- Cache  
- Event bus  

Separate:
- Read vs Write paths  

---

## рҹҡ« When NOT to Use This Design

- Small applications  
- Low traffic systems  
- Single-region systems  

---

## рҹ“Ң Real-World Flow Example

User creates an order:

1. Frontend вҶ’ API  
2. Backend validates + idempotency check  
3. SAP creates order  
4. Event emitted  
5. Other systems consume
